Q (User):
When I've looked at iced in the past trying to obtain some of this info, I
didn't see any way to query for it. Therefore, I'm assuming we would need to
add features to iced. Take for instance the current container.rs file from
iced. There are no get..() functions and the gui normally cycles a couple of
times to make final adjustments before rendering. Could you give me an idea
of how this would work for the container?


A (Assistant):
Looking at container.rs confirms what you suspected: Iced widgets are
essentially stateless descriptors that get re-built every frame. There are no
getters because the widget you hold (Container { padding, width, height, ... })
doesn't *know* its computed size — that gets calculated externally during the
layout pass and thrown away after rendering. So yes, exposing this requires
changes, but they're additive and don't fight Iced's architecture.

1. Understand where the data already exists (and dies)

Each frame Iced does roughly:
  1. view() builds the Element tree (your Container { ... } structs)
  2. Widget::layout(...) walks the tree → produces a layout::Node tree (this
     *has* x, y, width, height, padding-resolved bounds, child nodes)
  3. Widget::draw(...) walks both trees together and renders
  4. The Node tree is dropped

The "couple of cycles to settle" you mention is normally one layout pass per
event/redraw — Iced doesn't iteratively relax like a constraint solver, but
Length::Fill, Shrink, text measurement, and scrollables can cause a second
layout when content changes. The key point: all the geometry you want already
exists in layout::Node — it's just not exposed.

2. Two clean ways to expose it

Option A — Use Iced's existing Operation mechanism (preferred, minimal change)

Iced already has widget::Operation — a visitor that walks the widget tree with
access to layout. It's how scrollable::snap_to, text_input::focus, etc. work.
You add a new operation kind like Operation::QueryBounds(Id) or
Operation::Snapshot, and each widget implements it in its operate() method.

For Container, you'd:
  1. Already has id: Option<widget::Id>  ✅
  2. In Widget::operate(), when the operation is QueryBounds and the id
     matches, write layout.bounds(), padding-resolved inner bounds, alignment,
     and child count into the operation's output buffer.
  3. Recurse into children so nested containers report too.

Result: from Python you call something like ipg.query_layout(widget_id) →
returns a struct with { bounds, content_bounds, padding, alignment,
children: [...] }. No re-architecture, no new render pipeline.

Option B — Layout cache / snapshot side-channel

Store a HashMap<widget::Id, ResolvedLayout> populated during the layout pass
and made available to the host (your Python bridge). Every widget that has an
id contributes its computed Rectangle plus type-specific metadata.

This is heavier (touches every widget's layout() method or wraps it) but gives
you a complete describe_layout() in one shot rather than per-id queries.

I'd combine them: Option A for individual queries, Option B for the full-tree
snapshot.

3. Concrete sketch for Container specifically

What you'd add to Container:

    // New: optional intent metadata for AI/automation
    role: Option<String>,           // "primary_action", "toolbar", ...
    anchor: Option<Anchor>,         // TopLeft, Center, Right, Fill...
    tag: Option<String>,            // free-form group label

A new struct returned by queries:

    pub struct ContainerLayoutInfo {
        pub id: widget::Id,
        pub bounds: Rectangle,            // outer
        pub content_bounds: Rectangle,    // inner (after padding)
        pub padding: Padding,
        pub horizontal_alignment: alignment::Horizontal,
        pub vertical_alignment: alignment::Vertical,
        pub clipped: bool,
        pub free_space: Size,             // content_bounds - child bounds union
        pub child_ids: Vec<widget::Id>,
        pub role: Option<String>,
        pub anchor: Option<Anchor>,
    }

Inside Widget::operate() for Container:

    fn operate(&self, tree, layout, renderer, operation) {
        if let Some(id) = &self.id {
            operation.custom(Some(id), layout.bounds(), &mut ContainerProbe {
                id: id.clone(),
                outer: layout.bounds(),
                inner: layout.bounds().shrink(self.padding),
                padding: self.padding,
                h_align: self.horizontal_alignment,
                v_align: self.vertical_alignment,
                clipped: self.clip,
                role: self.role.clone(),
                anchor: self.anchor,
            });
        }
        // recurse into content as today
        self.content.as_widget().operate(
            &mut tree.children[0],
            layout.children().next().unwrap(),
            renderer,
            operation,
        );
    }

Your Python-facing layer collects probe outputs into a list and returns them.

4. Free-space / collision (the most useful AI signal)

Once every widget reports bounds keyed by id, you don't need each widget to
compute collisions itself. Compute it once at the host level after a snapshot:
  1. Free space per container = content_bounds minus union of child bounds
  2. Overlap = pairwise rectangle intersection on siblings
  3. Clipping = child bounds not fully inside parent content_bounds

This keeps the per-widget changes tiny and centralizes the "design check"
logic in icedpygui where you can iterate on it without forking iced widgets.

5. Handling the "couple of cycles" issue

Don't answer queries from inside view() or layout() — answer them from the
*last completed* layout. Cache the most recent layout::Node tree (or a
flattened id→bounds map) at the end of each frame. Queries from Python then
read that cache. This guarantees:
  1. You always get post-settled values
  2. Queries never trigger re-layout
  3. If the user resizes, the next frame updates the cache and emits a diff
     event

6. Minimum viable change for Container only

If you wanted a proof-of-concept before touching every widget:
  1. Add role, anchor, tag builder methods to Container
  2. Add a QueryBounds operation
  3. Implement it in Container::operate()
  4. Maintain a last_layout: HashMap<Id, Rectangle> updated each frame
  5. Expose ipg.get_bounds(id) and ipg.describe_container(id) to Python

That alone would let you prototype the "AI places by intent, then verifies"
loop on a single widget type, prove the ergonomics, then roll the same
pattern out to Row, Column, Grid, Stack, etc.

Bottom line

You don't need to rewrite Iced — the geometry already exists in layout::Node
every frame. You need:
  1. Stable ids on every widget (Container already has it; many don't)
  2. A query operation each widget implements
  3. A per-frame layout cache in icedpygui
  4. Optional intent metadata (role, anchor) for placement-by-meaning

That's the smallest change that turns Iced from a black-box renderer into
something an LLM can actually reason about.
